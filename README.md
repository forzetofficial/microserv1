# \## Задача 1: API Gateway

# 

# \### Сравнительная таблица возможностей решений для API Gateway

# 

# | Возможность | NGINX | Kong | Traefik | AWS API Gateway |

# |-------------|-------|------|---------|------------------|

# | Маршрутизация запросов на основе конфигурации | Да (конфиг-файлы) | Да (админ API + декларативная конфигурация) | Да (динамическая, из провайдеров) | Да (через AWS Console/API) |

# | Проверка аутентификации (JWT, OAuth, Basic и др.) | Частично (через Lua или модули) | Да (плагины: JWT, Key Auth, OAuth и др.) | Да (Middleware: ForwardAuth, JWT) | Да (Lambda authorizer, IAM, Cognito) |

# | Терминация HTTPS | Да (полноценная) | Да (встроенная) | Да (автоматические сертификаты через ACME) | Да (через CloudFront или пользовательский домен) |

# | Динамическое обновление правил | Нет (требуется reload) | Да (через REST API) | Да (автоматическое) | Да (через API) |

# | Встроенное rate limiting | Да (частично) | Да (плагин) | Да (встроенный middleware) | Да |

# | Логирование и мониторинг | Да (access log, метрики stub\_status) | Да (плагины, Prometheus) | Да (метрики, трассировка) | Да (CloudWatch, X-Ray) |

# | Сложность эксплуатации | Средняя | Средняя (нужна БД для кластера) | Низкая | Высокая (управляемый сервис) |

# | Бесплатность / стоимость | Open Source | Open Source (Enterprise — платно) | Open Source | Платно за запросы |

# 

# \### Выбор решения

# 

# \*\*Выбрано решение: Kong\*\*

# 

# \*\*Обоснование\*\*:

# 1\. \*\*Маршрутизация на основе конфигурации\*\* — Kong позволяет гибко настраивать маршруты через административное API или декларативные файлы (declarative config).

# 2\. \*\*Проверка аутентификации\*\* — из коробки поддерживаются популярные методы аутентификации через плагины (JWT, OAuth, Key Auth, LDAP и т.д.), что покрывает требование.

# 3\. \*\*Терминация HTTPS\*\* — Kong поддерживает termination TLS на уровне сервера, можно настроить сертификаты и SNI.

# 4\. Кроме того, Kong обладает высокой производительностью (на базе OpenResty/NGINX), поддерживает плагины на Lua/Python/Go, легко масштабируется в кластере с использованием PostgreSQL или Cassandra.

# 5\. По сравнению с чистым NGINX удобнее динамическое обновление маршрутов без перезагрузки. По сравнению с Traefik — лучше подходит для сложной политики безопасности (аутентификация, ACL). По сравнению с AWS API Gateway — работает в любом окружении (не только AWS) и имеет открытый код.

# \## Задача 2: Брокер сообщений

# 

# \### Сравнительная таблица возможностей брокеров сообщений

# 

# | Возможность | RabbitMQ | Apache Kafka | Redis (с модулем Streams) | NATS (с JetStream) |

# |-------------|----------|--------------|----------------------------|---------------------|

# | Кластеризация для надёжности | Да (кластер очередей, Quorum Queues) | Да (кластер брокеров, репликация партиций) | Да (кластер Redis Sentinel или Cluster) | Да (кластер NATS с RAFT) |

# | Хранение сообщений на диске в процессе доставки | Да (очереди могут быть durable) | Да (по умолчанию на диске, retention) | Частично (только если включить AOF и RDB) | Да (JetStream — на диске) |

# | Высокая скорость работы | Средняя (~10–30K msg/sec) | Очень высокая (~100K+ msg/sec) | Очень высокая (в памяти) / ниже при записи на диск | Высокая (NATS — очень быстрый, JetStream добавляет задержку) |

# | Поддержка различных форматов | Любые (payload — массив байт) | Любые (сериализация через Avro, JSON, Protobuf — с помощью Schema Registry) | Любые (строки/байты) | Любые (массив байт) |

# | Разделение прав доступа к потокам | Да (виртуальные хосты, пользователи, ACL) | Да (ACL по темам, SCRAM, OAuth) | Частично (пароли, но ACL на уровне команд ограничен) | Да (NGS, permissions по субъектам) |

# | Простота эксплуатации | Средняя (требует контроля очередей, памяти) | Сложная (требует ZooKeeper/KRaft, настройки репликации, балансировки) | Простая (один бинарник, много режимов) | Средняя (NATS core прост, JetStream требует понимания) |

# 

# \### Выбор решения

# 

# \*\*Выбрано решение: Apache Kafka\*\*

# 

# \*\*Обоснование\*\*:

# 1\. \*\*Кластеризация\*\* — Kafka изначально спроектирована для распределённой работы, поддерживает репликацию партиций, лидеров, восстановление при сбоях.

# 2\. \*\*Хранение на диске\*\* — все сообщения сохраняются на диск с настраиваемым retention, что гарантирует доставку и возможность повторного чтения.

# 3\. \*\*Высокая скорость\*\* — Kafka обеспечивает производительность до сотен тысяч сообщений в секунду на кластер, оптимизирована для потоковой обработки.

# 4\. \*\*Различные форматы\*\* — сообщения не накладывают ограничений на формат, а с использованием Schema Registry можно управлять схемами (Avro, JSON, Protobuf).

# 5\. \*\*Разделение прав доступа\*\* — поддерживаются ACL на чтение/запись к темам, аутентификация (SASL, mTLS), интеграция с RBAC (например, через Kafka RBAC в конвейерах).

# 6\. \*\*Простота эксплуатации\*\* — несмотря на относительную сложность, Kafka имеет огромное сообщество, инструменты (Kafka Manager, Cruise Control, Prometheus exporter) и операторские практики (Strimzi, Confluent Platform).

# 

# \*\*Почему не RabbitMQ\*\*: ниже скорость при высоких нагрузках, сложнее масштабировать очереди на много потребителей без потери порядка и производительности.

# 

# \*\*Почему не Redis\*\*: не гарантирует durable хранение сообщений при сбоях, ACL на потоки ограничены, основное назначение — кэш.

# 

# \*\*Почему не NATS\*\*: хотя NATS с JetStream очень интересен и быстр, он менее распространён в enterprise-практиках, сообщество меньше, меньше готовых интеграций с системами мониторинга и обработки.
